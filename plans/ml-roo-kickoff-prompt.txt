I am building a SillyTavern extension called "Memory Loom" (ML). I have an approved UI mockup
(memory-loom-mockup.html) that shows the exact layout, components, and interactions required.
Please treat it as the visual blueprint — replicate its structure faithfully.

---

OVERVIEW

Memory Loom is a SillyTavern extension that acts as a highly specialized, curated memory archive
for long-running roleplay. It is not a generic RAG tool. Its purpose is to preserve what changed,
organize what should carry forward, and inject contextually relevant memory into the system prompt
automatically.

The extension works on three layers:

1. WRITE — When the user closes a scene, a Memory Writer LLM generates memory entries covering
   all narratively, emotionally, and psychologically significant moments in the scene. The user
   reviews and approves before anything is committed.

2. RETRIEVE — On every message, a lightweight Keyword Sidecar LLM extracts themes and character
   references from recent context and feeds them as a clean query to an Embedding model. The
   embedding model matches against the memory bank by similarity score and injects matched entries
   silently into the system prompt.

3. CONSOLIDATE — On user request, a Consolidation LLM combines selected memories and scene
   summaries into a higher-level carry-forward summary, lowering the injection priority of the
   source memories it covers without deleting them.

Memory Loom should provide contextual judgment, not narrative direction. It answers:
  What happened? What changed? Who was affected? What context should guide the model going forward?
It does not answer: What should happen next? What plot hooks should be pursued?

DATA STRUCTURES

Memory Entry:
{
  id: string,
  title: string,
  datetime: string,                    // narrative date/time string, e.g. "Heian Era · ~1104 CE · dusk"
  content: string,                     // full narrative prose body
  primaryCharacter: string,            // single character name (determines folder routing)
  primaryCharacters: string[],         // array — populated when 2+ primaries (routes to Group)
  keyCharacters: string[],             // supporting cast, reference only, no routing effect
  category: "character" | "world" | "plot",
  folderId: string,                    // ID of the folder/subfolder this entry lives in
  tags: string[],                      // descriptive metadata for library browsing and filtering
                                       // NOTE: tags are for human use only. They are NOT used in
                                       // the embedding pipeline. The sidecar generates its own
                                       // ephemeral keyword queries from chat context at runtime.
  status: "active" | "consolidated" | "archived" | "pinned" | "superseded",
  delta: {
    before_state: string,              // what was true before this memory
    after_state: string,               // what meaningfully changed after
    delta: string,                     // short plain-language description of the change
    delta_type: string[],              // one or more tags from the delta type vocabulary (see below)
    low_delta_flag: boolean            // true if memory writer flagged this as a weak delta
  },
  vectorId: string | null,             // reference to stored embedding vector
  stickiness: number,                  // messages to stay injected after firing (0 = global default)
  cooldown: number,                    // messages before entry can fire again (0 = global default)
  sceneId: string | null,             // scene that generated this entry, null if manual
  source: "llm_generated" | "manual",
  createdAt: number,                   // timestamp
  updatedAt: number
}

Delta Type Vocabulary (fixed list — do not make configurable):
  relationship_shift, knowledge_change, secret_created, secret_revealed, secret_concealed,
  promise_created, threat_created, trust_change, suspicion_change, character_state_change,
  world_state_change, faction_state_change, plot_progression, conflict_escalation,
  conflict_resolution, item_state_change, location_state_change, status_change,
  self_restraint, public_alignment

Folder:
{
  id: string,
  name: string,
  type: "world" | "characters" | "plot" | "primary" | "subfolder",
  parentId: string | null,            // null = top-level folder
  characterName: string | null,       // set when this is a character subfolder
  hasImage: boolean,                  // whether a banner image is set (character subfolders only)
  imagePath: string | null,           // path to cropped banner image
  entryCount: number,
  createdAt: number
}

Scene:
{
  id: string,
  status: "open" | "closed",
  messageStart: number,
  messageEnd: number | null,
  llmSummary: string,                 // INTERNAL ONLY — never injected into main ST prompt
  consolidatedInto: string | null,    // consolidation entry ID if this scene was consolidated
  createdAt: number
}

Consolidation Entry (stored in Plot folder):
{
  id: string,
  title: string,
  type: "character_consolidation" | "plot_subarc_consolidation" | "arc_consolidation" |
        "folder_consolidation" | "mixed_consolidation",
  scope: {
    folders: string[],                // folder IDs included
    sourceIds: string[]               // entry and scene IDs used as source
  },
  timeRange: string,                  // narrative time period covered
  before_state: string,
  after_state: string,
  summary: string,                    // full consolidation text for library review
  preferred_injection: string,        // compact version injected into prompt
  carry_forward_context: string[],    // neutral state facts (present tense, no future direction)
  key_changes: string[],
  character_impact: string[],
  relationship_impact: string[],
  plot_impact: string[],
  world_impact: string[],
  source_memories: string[],          // entry IDs
  status_updates: [                   // status changes to apply to source entries on commit
    { sourceId: string, newStatus: string }
  ],
  tags: string[],
  status: "active" | "archived" | "pinned",
  vectorId: string | null,
  stickiness: number,
  cooldown: number,
  createdAt: number,
  updatedAt: number
}

Settings:
{
  enabled: boolean,
  connections: {
    memoryWriterLLM: string,
    consolidationLLM: string,
    sidecarLLM: string,
    embeddingModel: string
  },
  scanFrequency: number,              // default: 1 (every message)
  injection: {
    enabled: boolean,
    placement: "above_card" | "below_card" | "top" | "bottom",
    maxEntriesPerMessage: number      // default: 3
  },
  vectorization: {
    similarityThreshold: number,      // default: 0.75
    querySource: "keywords" | "raw",
    raw: {                            // only used when querySource is "raw"
      scanDepth: number,              // default: 10
      chunkSize: number,              // default: 256
      overlapTokens: number,          // default: 32
      topK: number,                   // default: 10
      distanceMetric: "cosine" | "dot_product" | "euclidean",
      rerank: boolean
    },
    defaultStickiness: number,        // default: 0 (no stickiness)
    defaultCooldown: number           // default: 0 (no cooldown)
  },
  memoryWriting: {
    folderSuggestions: boolean,       // default: false
    sceneSummaryPrompt: string,
    memoryEntryPrompt: string
  },
  decay: {
    enabled: boolean,                 // default: false
    mode: "linear" | "exponential" | "step",
    decayStart: number,               // scenes before decay begins, default: 5
    minimumPriority: number,          // floor, default: 0.3
    exemptPinned: boolean             // default: true
  }
}

CORE BEHAVIORS TO IMPLEMENT

1. MESSAGE ACTION BUTTONS — THREE STATES
   Memory Loom adds a single action button to every message in ST's chat UI.
   The button has three states and cycles based on scene state:

   fa-book-open (unscanned):
     Default state for all new messages. Pressing it opens a scene starting at this message.

   fa-feather (scene open):
     Shown on all messages within the currently open scene. Pressing it closes the scene
     and triggers the memory writer flow.

   fa-book (scanned/locked):
     Shown on messages that have been processed by a completed scene or batch scan.
     Cannot be pressed. Locked permanently unless the user runs Undo Last Scan.

   IMPORTANT: Use inline SVG icons, NOT Font Awesome CDN. FA CDN is unreliable in ST.
   Copy the SVG icon definitions exactly from memory-loom-mockup.html.

2. KEYWORD SIDECAR PIPELINE (runs every message by default)
   - After each new message, call the Keyword Sidecar LLM with recent context
   - Sidecar returns a short list of themes, character names, and event references
   - These keywords are fed as a query to the embedding model
   - Embedding model matches keywords against all stored entry vectors
   - Entries above the similarity threshold are candidates for injection
   - Apply stickiness and cooldown rules (see Section 7) before final injection decision
   - Inject matched entries silently into ST system prompt — user sees no visual feedback
   - The sidecar runs silently. No UI indicator. Only the memory writer flow has visible feedback.

   CRITICAL DISTINCTION: Entry tags (stored on each entry) are for library browsing only.
   They are NOT used in the embedding pipeline. The sidecar generates its own ephemeral
   keyword queries from live chat context at runtime. These are two completely separate systems.

3. SCENE MANAGEMENT
   - Scenes open when the user presses a fa-book-open button on any message
   - Scenes close when the user presses the fa-feather button
   - Scene open records the starting message index
   - Scene close records the ending message index and triggers the memory writer flow
   - Only one scene can be open at a time
   - Open scene state persists across page reloads

4. MEMORY WRITER FLOW (on scene close)
   Step 1 — Scene Summary:
     Call Memory Writer LLM with the scene's messages plus all previous closed scene summaries
     as context. Generate a rich internal scene summary covering: key events, emotional turning
     points, psychological shifts, characters present, unresolved tensions, and relationship
     developments. This summary is INTERNAL ONLY — never injected into the main ST prompt.
     Store it in the Scenes view, user-editable.

   Step 2 — Entry Generation:
     Using the scene summary as primary source, call Memory Writer LLM to identify all
     narratively, emotionally, or psychologically significant moments and generate one memory
     entry per moment.

     Each entry includes:
       - title, datetime, content (full prose narrative body)
       - primaryCharacter / primaryCharacters, keyCharacters
       - delta block: before_state, after_state, delta, delta_type[], low_delta_flag
       - suggested tags (descriptive, for library use only)
       - suggested folder destination

     NEVER generate entries for {{user}} / the user persona. Only NPCs and {{char}}.

     Entries where the delta is essentially null (nothing meaningfully changed) should be
     flagged with low_delta_flag: true and shown to the user with a "low delta" warning badge.
     Do NOT auto-discard them. The user makes the final decision.

   Step 3 — Pending Review:
     Show all proposed entries on the Home tab as collapsible cards.
     Nothing commits until the user approves.
     Per-entry actions: Commit, Regenerate (with optional guidance prompt), Edit, Discard.
     Global actions: Commit All, Discard All.

   Step 4 — On Commit:
     Auto-route the entry to its destination folder:
       - Single primary character → that character's subfolder under Characters
         (create the subfolder automatically if it doesn't exist)
       - 2+ primary characters → Group subfolder under Characters
       - World or Plot category → user-selected folder
     Auto-embed the entry through the embedding model before storing.
     Apply suggested status updates.

5. MANUAL ENTRY CREATION
   - User fills in: Title, Date/Time, Content, Category, Folder, Primary Character(s), Key Characters
   - Auto-tag assist button available: calls Memory Writer LLM to suggest tags from the content
     (optional, user can skip)
   - On save: auto-embed, auto-route same as LLM-generated entries
   - All fields have expand popout buttons (editor_maximize pattern from NWST/RST — see UI Notes)

6. FOLDER SYSTEM
   Top-level folders (always present):
     World — general lore, world events, setting info affecting multiple characters
     Characters — contains character subfolders
     Plot — story arcs, major turning points, consolidation entries

   Character subfolders (under Characters):
     One per character. Named after the character. Has image upload + new entry buttons.
     Auto-created when an entry with a new primary character is committed.
     Group subfolder — auto-created for entries with 2+ primary characters.
     Group does NOT get an image upload button.

   User-created Primary folders:
     Sit at the same level as World / Characters / Plot.
     Named by the user. Get a new entry button only.

   User-created Subfolders:
     Can be created under any folder.
     Under Characters → get image upload + new entry buttons.
     Under World, Plot, or user primary folders → get new entry button only.

   Character Banner Images:
     Uploaded per character subfolder. Wide rectangular crop (banner aspect ratio).
     Crop dialog uses aspect-locked drag-to-reposition with corner resize handles.
     Crop supports both mouse and touch events (see UI Notes).
     Banner is hidden entirely when no image is set — no placeholder block.
     Displayed above the character name row when set, full width, color, brightness 0.92.

7. STICKINESS AND COOLDOWN
   Once an entry fires (is injected), stickiness keeps it injected for N subsequent messages
   even if the similarity score drops below threshold. Prevents flickering.

   After the stickiness window expires, cooldown prevents the entry from firing again for N
   messages. Prevents a single memory dominating every injection slot.

   Global defaults are set in Settings → Vectorization.
   Per-entry overrides are configurable in the expanded entry view in the library.
   An override value of 0 means "use global default."
   Stickiness and cooldown apply to consolidation entries as well as regular entries.

8. ENTRY STATUSES
   active:       Normal injection priority. Default for all new entries. No badge shown.
   consolidated: Lower default injection priority. A consolidation entry now carries its meaning.
   archived:     Searchable in library but never injected by default.
   pinned:       Always injects at full priority regardless of similarity score, stickiness, or decay.
   superseded:   Kept as historical record. A newer memory or consolidation overrides it.

   Status is set manually by the user or automatically by the consolidation commit process.
   Display as small monochrome badges on entry cards in the library (see mockup).
   Active entries show no badge — it is the default unmarked state.

9. CONSOLIDATION WORKFLOW
   User-directed. Never automatic.

   Three consolidation modes:
     Consolidate Selected: user manually picks specific entries and/or scene summaries.
     Consolidate Folder: user selects a folder; ML pulls all approved entries from it.
     Consolidate Mixed Scope: user picks multiple folders and entry types together.

   Flow:
     1. User selects sources in the consolidation workspace
     2. Call Consolidation LLM with selected entries + scene summaries + scope
     3. Generate consolidation draft (see Consolidation Entry data structure above)
     4. User reviews, edits, or regenerates
     5. On commit:
        - Store consolidation entry in Plot folder
        - Apply status_updates to source entries (typically: active → consolidated)
        - Auto-embed the consolidation entry
        - Consolidation entry becomes preferred injection source for broad recall

   IMPORTANT: Consolidation does not delete source memories. Source memories remain in the
   library and remain searchable. Their injection priority is reduced, not removed.

   The consolidation LLM must not include open_threads, future_questions, or suggested_next_steps
   in its output. Unresolved context must be expressed as present state, not future-facing hooks.
   Example of what NOT to do: "Will Sachiko forgive Rukia?"
   Example of correct approach: "Sachiko has not forgiven Rukia and remains guarded around her."

10. BATCH SCAN
    One-time operation on full chat history. Triggered by button in Settings → Data.
    Auto-detects scene boundaries. Creates blank character subfolders for unrecognized names.
    Generates scene summaries and initial memory entries per detected scene.
    Non-compounding — will not stack on existing data.
    Cannot be undone (Undo Last Scan is unavailable after batch scan, but remains visible).

11. MEMORY DECAY (optional, off by default)
    Gradually reduces injection priority of older entries over time.
    Configurable: mode (linear / exponential / step), decay start (scenes), minimum priority floor.
    Pinned entries are exempt from decay by default (configurable).
    This is separate from and lower priority than the status system.
    A pinned entry always injects regardless of decay.
    A consolidated entry's reduced priority applies on top of decay.

DATA STORAGE

All data is stored per-chat using ST's extension data storage API.
Memory Loom data does not persist across chats — it is specific to the chat it was built in.
Storage is organized by:
  entries/          — all memory entries
  folders/          — folder and subfolder definitions
  scenes/           — scene records and summaries
  consolidations/   — consolidation entries
  vectors/          — embedding vectors per entry (keyed by entry ID)
  settings          — user settings (global, not per-chat)

---

IMPORTANT BEHAVIORAL RULES

- NEVER generate memories for {{user}} or the user persona. Only NPCs and {{char}}.
  This rule must be hardcoded into the memory writer prompt and enforced in code.

- Entry tags are for library browsing only (human-facing). They are NOT fed to the embedding
  model. The sidecar generates its own keyword queries at runtime from live chat context.
  Do not conflate these two systems.

- Nothing commits to storage without user approval. The pending review flow on the Home tab
  is the only path to committed entries. Auto-commit is never acceptable.

- Low-delta entries are flagged, not discarded. The user decides.

- Scene summaries are NEVER injected into the main ST prompt under any circumstances.
  They are internal LLM notepads only, visible in the Scenes view.

- Consolidation output must not contain open_threads, future_questions, or suggested_next_steps.
  Express unresolved context as present state only.

- The Undo Last Scan button always remains visible in Settings → Data.
  Its subtext changes to "cannot undo a batch scan" after a batch scan, and the button is
  disabled — but it is never hidden.
